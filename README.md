# Chat Smart - Backend

A real-time one-on-one chat application backend built with Spring Boot. It provides REST APIs for authentication, user management, messaging, and conversation management, along with WebSocket support for real-time message delivery.

## Tech Stack

- **Java 21**
- **Spring Boot 4.0.1**
- **Spring Security** + **JWT** (stateless authentication)
- **Spring Data JPA / Hibernate** (ORM)
- **MySQL** (database)
- **Spring WebSocket** (real-time messaging)
- **SpringDoc OpenAPI** (Swagger UI for API docs)
- **Lombok** (boilerplate reduction)
- **Maven** (build tool)

## Prerequisites

- **Java 21** or later
- **MySQL** server running
- **Maven** (or use the included Maven Wrapper `./mvnw`)

## Getting Started

### 1. Clone the repository

```bash
git clone <repository-url>
cd be
```

### 2. Create the database

Create a MySQL database named `ChatSmart`:

```sql
CREATE DATABASE ChatSmart;
```

> Tables are auto-generated by Hibernate on startup (`ddl-auto=update`).

### 3. Configure environment variables

Copy the example env file and fill in your values:

```bash
cp .env.example .env
```

Edit `.env` with your configuration:

```properties
# Database
DB_USERNAME=root
DB_PASSWORD=your_password

# JWT config
JWT_SECRET=your_base64_encoded_secret_key
ACCESS_TOKEN_EXP=86400000
REFRESH_TOKEN_EXP=604800000
```

| Variable            | Description                                                               |
| ------------------- | ------------------------------------------------------------------------- |
| `DB_USERNAME`       | MySQL username                                                            |
| `DB_PASSWORD`       | MySQL password                                                            |
| `JWT_SECRET`        | Base64-encoded HMAC-SHA256 secret key for signing JWTs                    |
| `ACCESS_TOKEN_EXP`  | Access token expiration time in milliseconds (e.g. `86400000` = 24h)      |
| `REFRESH_TOKEN_EXP` | Refresh token expiration time in milliseconds (e.g. `604800000` = 7 days) |

### 4. Run the application

```bash
./mvnw spring-boot:run
```

The server starts on **http://localhost:8080**.

Swagger UI is available at **http://localhost:8080/swagger-ui**.

## API Endpoints

All REST responses are wrapped in a standard format:

```json
{
  "status": 200,
  "message": "...",
  "data": {}
}
```

### Authentication (`/api/v1/auth`)

| Method | Endpoint    | Auth | Description                  |
| ------ | ----------- | ---- | ---------------------------- |
| `POST` | `/register` | No   | Register a new user          |
| `POST` | `/login`    | No   | Login and receive JWT tokens |

### User (`/api/v1/user`)

| Method | Endpoint     | Auth | Description                                          |
| ------ | ------------ | ---- | ---------------------------------------------------- |
| `GET`  | `/profile`   | Yes  | Get current user's profile                           |
| `PUT`  | `/update`    | Yes  | Update profile (display name, gender, date of birth) |
| `POST` | `/find-user` | Yes  | Search users by display name                         |
| `GET`  | `/get/{id}`  | Yes  | Get a user by ID                                     |

### Messages (`/api/v1/messages`)

| Method | Endpoint              | Auth | Description                                          |
| ------ | --------------------- | ---- | ---------------------------------------------------- |
| `POST` | `/send`               | Yes  | Send a message (auto-creates conversation if needed) |
| `GET`  | `/with/{userId}`      | Yes  | Get all messages with a specific user                |
| `PUT`  | `/delete/{messageId}` | Yes  | Soft-delete a message                                |

### Conversations (`/api/v1/conversations`)

| Method | Endpoint                   | Auth | Description                                 |
| ------ | -------------------------- | ---- | ------------------------------------------- |
| `GET`  | `/get`                     | Yes  | List all conversations for the current user |
| `PUT`  | `/delete/{conversationId}` | Yes  | Soft-delete a conversation                  |

### WebSocket

| Protocol | Endpoint | Auth         | Description                |
| -------- | -------- | ------------ | -------------------------- |
| `WS`     | `/ws`    | Yes (cookie) | Real-time message delivery |

When a message is sent via the REST API, it is automatically pushed to both sender and receiver through the WebSocket connection.

**Payload format:**

```json
{
  "type": "MESSAGE",
  "data": { "...message fields..." }
}
```

## Authentication Flow

1. **Register** -- `POST /api/v1/auth/register` with username and password
2. **Login** -- `POST /api/v1/auth/login` returns an access token in the response body and sets a refresh token as an HttpOnly cookie
3. **Authenticated requests** -- Include the access token via `Authorization: Bearer <token>` header or `accessToken` cookie
4. **WebSocket** -- Authentication is handled during the handshake using the `accessToken` cookie

## Database Schema

Four tables are auto-generated by Hibernate (UUID primary keys, with `createdAt` and `updatedAt` timestamps):

- **users** -- `username`, `password`, `displayName`, `status`, `image`, `gender`, `dateOfBirth`
- **conversations** -- `user1_id`, `user2_id`, `deleteAtUser1`, `deleteAtUser2`
- **messages** -- `conversation_id`, `sender_id`, `receiver_id`, `content`, `displayOrder`, `isDeleted`
- **messages_status** -- `message_id`, `status` (SENDING / SENT / RECEIVED / WATCHED)
